# F-RID ENHANCEMENT ANALYSIS: T810A1-NOTE-001 (T810A2 Schema Handoff)

## I. EXECUTIVE SUMMARY

**Purpose**  
To analyze `T810A1-NOTE-001 (T810A2 Schema Handoff)` and associated T810A2 artifacts to understand how Stable/Dynamic schema decisions should influence `T810A1-PROMPT` Integration Notes (INT) and Constraints (CON), particularly for Story `T810A1-S05 (Execution Protocol)` and Block 4 schema references.

**Scope**  
- Focus: JSON schema architecture, aggregation pattern, and field classification decisions that affect how PROMPT describes and uses patient data.  
- Categories impacted: existing and new `T810A1-INT-*` items plus a new constraint `T810A1-CON-004` mirroring fixed‑keys behavior, as indicated in the Phase 0 plan.  
- Sources: handoff brief, `request_T810A2-SCHEMA.md`, `template_dynamic_tracking_schema.yaml`, `field_classification_mapping.md`, and aggregation examples.

**Method**  
1. Summarize key schema deliverables communicated in the handoff brief.  
2. Map those deliverables to candidate `T810A1-INT-*` and `T810A1-CON-*` items (IDs only) that will later be specified in the Request.  
3. Identify alignment gaps between PROMPT and SCHEMA artifacts and highlight questions to resolve before editing F-RIDs.

**High-Level Outcome (Intent Only)**  
- Confirms **Stable vs Dynamic architecture**, **single‑entry model**, **fixed keys**, **mandatory/optional field classification**, and **Aggregation Pattern A** as the working assumptions for S05.  
- Identifies candidate enhancements to `T810A1-INT-001`, `T810A1-INT-003`, `T810A1-INT-005`, and a new `T810A1-INT-006` plus `T810A1-CON-004`, ensuring PROMPT respects T810A2 schema governance.  
- Surfaces a small set of integration questions (e.g., handling optional fields and cross‑feature vocabulary drift) for Phase 1.

## II. SOURCE MATERIAL SUMMARY (T810A1-NOTE-001)

### A. Source Overview

- **Artifact**: `handoff_brief_T810A2-SCHEMA_to_T810A1-PROMPT_phase2.md`.  
- **Role**: Communicates MVP Stable SCHEMA, Dynamic SCHEMAs, aggregation patterns, and integration touchpoints from `T810A2` to `T810A1` for Execution Protocol design.

### B. Key Deliverables (Per Phase 0 Plan)

1. **Stable/Dynamic Schema Architecture**  
   - Stable JSON: patient profile, **patient‑managed and read‑only** for LLM_Gastro (per `T810A2-CON-004`).  
   - Dynamic JSON: per‑interaction tracking entries (e.g., meal, stool, digestion, mental) generated by the agent.

2. **Single‑Entry Model**  
   - Each interaction results in **exactly one Dynamic JSON entry**, not a cumulative object.  
   - This simplifies S05 wording and aggregation logic (append‑only behavior).

3. **Fixed Keys Constraint**  
   - Per `T810A2-CON-005`, Dynamic JSON keys are **fixed and schema‑defined**; the agent SHALL NOT invent new keys at runtime.  
   - PROMPT must reflect this fixed‑key discipline when instructing LLM_Gastro.

4. **Mandatory/Optional Field Classification**  
   - `field_classification_mapping.md` defines mandatory vs optional fields for required entry types (meal, stool, digestion, mental).  
   - These classifications are intended to drive **Probe triggering logic** (e.g., missing mandatory fields → questions).

5. **Aggregation Pattern A (Recommended)**  
   - Dynamic entries are aggregated in a **single mixed chronological array** (Pattern A), with each element carrying an `entry_type`.  
   - Alternatives (per‑type arrays, hybrid indices) exist but are not the recommended MVP baseline.

6. **Combined Dynamic SCHEMA Block**  
   - `template_dynamic_tracking_schema.yaml` provides an LLM‑facing `dynamic_schemas` block consolidating per‑type structures.  
   - S05 and Block 4 are expected to reference this as the **canonical schema snippet** for Dynamic entries.

## IV. GAP ANALYSIS & ALIGNMENT RISKS

### A. Coverage Strength

- Strong alignment on:
  - The overall Stable/Dynamic separation and patient‑managed profiles.  
  - The fixed‑keys approach to Dynamic JSON.  
  - Mandatory/optional fields as a driver for Probe behavior.  
  - Pattern A as the default aggregation strategy for MVP.

- Moderate clarity on:
  - How optional fields should influence Probe intensity (e.g., when to ask vs when to accept null).  
  - How PROMPT should behave if user input suggests an entry type not yet fully specified (e.g., sleep/medication skeleton entries).

### B. Identified Gaps

1. **Optional Fields & User Experience**  
   - The handoff specifies optional fields but does not fully describe how aggressively the agent should **probe to fill** them.  
   - Over‑probing could reduce usability; under‑probing could reduce clinical richness.

2. **Vocabulary & Future Evolution**  
   - T810A2 governs controlled vocabularies (per `T810A-ADR-002` and T810A2 NFRs), but PROMPT must stay robust to **vocabulary updates**.  
   - Current materials do not fully define how PROMPT should react if vocabularies expand or deprecate values.

3. **Aggregation Alternatives & Future Phases**  
   - Pattern A is recommended now, but the presence of Patterns B and C suggests potential future shifts (e.g., for reporting).  
   - PROMPT’s INTs should be written to **tolerate** future aggregation changes without complete rework.

### C. Risk Summary (If Unaddressed)

- Inconsistent alignment between PROMPT and SCHEMA could lead to:
  - Dynamic JSON that **violates fixed‑keys constraints** or ignores mandatory field mapping.  
  - Confusing or contradictory instructions about how many entries to generate or how to aggregate them.  
  - Increased rework in later phases if INTs are not written with **forward compatibility** in mind.

## V. OPEN QUESTIONS & RECOMMENDATIONS FOR PHASE 1

### A. Questions for Client / Governance

1. **Probe Aggressiveness for Optional Fields**: Should the agent **occasionally suggest** optional fields (e.g., “If you remember, you can add notes about sleep”) or generally stay silent unless clearly relevant?  
2. **Tolerance for Schema Evolution**: Is it acceptable for PROMPT to mention that schemas may evolve and that the agent will **follow the latest project knowledge**, or should such evolution remain implicit?  
3. **Aggregation Pattern Longevity**: Does the Client view Pattern A as a **long‑term choice** or primarily as an MVP compromise, and should INT wording reflect this?

### B. Recommendations for Phase 1 Editing

- When updating `T810A1-INT-*` items:
  - Explicitly reference `template_dynamic_tracking_schema.yaml` and `field_classification_mapping.md` as authoritative for S05’s schema assumptions.  
  - Make the single‑entry model and fixed‑keys behavior **highly visible** so that future prompt refactors do not accidentally regress.  
  - Document Probe behavior in INT/IG such that mandatory field gaps **trigger clarification**, while optional fields are treated as enrichment.

- When adding `T810A1-CON-004`:
  - Keep wording short and focused on PROMPT‑level obligations (e.g., “shall only use keys defined in T810A2 schemas”).  
  - Avoid duplicating T810A2 constraint details; instead, reference `T810A2-CON-005` by ID.
