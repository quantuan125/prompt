# COMMUNICATION: T104 -> T103 — Claude Code Skill External-Review Execution Reliability Issue

## I. PURPOSE

This communication informs the T103 skill owner of a concrete execution reliability issue encountered while AC001.6 attempted to use the local Claude Code skill for a delegated consultant-side `external_review` step.

The issue is not that Claude Code is unusable. The issue is that the current skill guidance does not provide a dependable automation-safe path for a bounded "directly author exactly one artifact" workflow in this environment, and the observed runtime behavior diverged from the expected orchestration contract.

The immediate client impact is that the manual Claude execution step had to be handed back to the client instead of being completed inside the orchestration loop.

## III. OBSERVED ISSUE

### A. Expected Behavior

The expected behavior was:
1. invoke Claude Code from Codex CLI,
2. have Claude read the specified AC001.6 inputs,
3. create exactly one analysis artifact at the target path,
4. return a concise terminal summary,
5. allow orchestration to continue without manual intervention.

### B. Actual Behavior

The actual behavior was:
1. bounded non-interactive `claude -p` runs returned no useful output,
2. the target artifact was not created during those runs,
3. there was no surfaced stderr explaining why the write did not occur,
4. interactive TTY runs showed a workspace trust / safety confirmation prompt,
5. that trust prompt blocked automation and appears to require per-session handling,
6. even after confirmation in an interactive run, the expected bounded artifact write did not complete deterministically inside the orchestration window,
7. the workflow had to fall back to a consultant-owned/manual path rather than a clean skill-driven direct-authoring flow.

### C. Net Effect

The current Claude Code skill contract overstates practical reliability for this specific usage pattern:
- direct repo authoring,
- single-file bounded write,
- Codex-mediated execution,
- WSL/bash environment,
- non-interactive or lightly interactive automation.

## V. ROOT ISSUE ASSESSMENT

This looks like a skill/runtime integration problem rather than a content problem.

### A. Likely contributing factors

- The skill guidance does not adequately surface the workspace trust prompt as an automation blocker.
- The documented print-mode posture is not reliable for direct file authoring in this environment.
- The guidance does not clearly distinguish:
  - "good for read-only one-shot output"
  - vs "good for deterministic bounded file authoring"
- The current skill does not provide a hardened fallback sequence when direct authoring fails.

### B. Why this matters

The client is using Claude Code as part of a governed multi-agent orchestration loop. In that workflow, silent or quasi-silent failure is worse than an explicit rejection because it leaves the consultant uncertain whether:
- Claude is still waiting for trust confirmation,
- Claude is waiting on a hidden edit permission branch,
- the command has ended but the artifact was never written,
- or the skill contract itself is mismatched to the actual runtime.
