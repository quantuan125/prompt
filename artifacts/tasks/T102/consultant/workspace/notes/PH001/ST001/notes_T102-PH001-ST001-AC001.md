#### B. Narrative Summary (Minutes-Style)
The client approved **Model C (Standards-Doc Migration)** as the authoritative “where-spec-lives” model for Phase 1. The approval is grounded in the goal of transforming `T102C (CONCEPT)` into a dedicated dynamic working index that supports SSOT artifacts across all scopes while avoiding context-window overload. As part of this direction, Concept should not embed detailed content (including ADR bodies and their detailed `CLAUSE` specifications). Instead, Concept becomes the primary navigation surface (registers + indexes + canonical links).

The session also confirmed that Phase 1 should avoid introducing a new token (e.g., `SPEC`) unless it is proven necessary. The default traceability mechanism for adopted specs is therefore “canonical path + anchor” surfaced via the future Specs Index standard (AC002). This preserves alignment with the existing standards adoption design (`STD` adopts exactly one canonical spec), while containing governance-change risk in Phase 1.

### Session — 2026-02-04 (Model D Pivot: Combined ADR+Spec Files)

**Participants**: LLM_Consultant, Client
**Primary focus**: Client-initiated pivot from Model C to Model D; record rationale, confirm design decisions, and update downstream activity implications.

#### A. Agenda / Topics
1. Assess Model C implementation complexity and file proliferation concerns.
2. Evaluate Model D (Combined ADR+Spec Files) as a pragmatic Phase 1 alternative.
3. Confirm Model D design decisions: path convention, extraction scope, file naming, directory structure.
4. Update AC002/AC003 task scope to align with Model D.

#### B. Narrative Summary (Minutes-Style)
The client raised concerns that Model C implementation would result in excessive file proliferation (separate ADR files + separate spec files) and increased token/context load when agents need to understand both the decision rationale and the specification. The client proposed **Model D (Combined ADR+Spec Files)** as a pragmatic refinement that:

1) Preserves the core Phase 1 goal: **move ADR bodies out of Concept** to reduce bloat.
2) Combines each ADR body with its embedded CLAUSE specifications into a **single dedicated file** per ADR.
3) Avoids the dual-path complexity of Model C (no separate `adrs/` and `specs/` directories).
4) Maintains full compatibility with existing `CLAUSE` semantics (CLAUSE remains ADR-internal).

The consultant provided independent analysis grounded in industry best practices (ADR guidelines from adr.github.io, AWS, MADR). While industry guidance recommends "lean ADRs" and separation of detailed specs, the consultant assessed that **Model D is viable for Phase 1** given:
- Single ownership model (LLM_Consultant owns both decision and spec currently)
- Lower migration cost than Model C
- Easy future separation if ownership diverges in Phase 2

The client confirmed the following design decisions:
- **Path convention**: `prompt/artifacts/tasks/T102/consultant/standards/` (specs-first framing)
- **Extraction scope**: ALL ADRs (Initiative + Epic + Feature levels)
- **File naming**: `T102-ADR-###_<title-slug>.md` pattern (e.g., `T102-ADR-004_decision-records-index.md`)
- **Directory structure**: Flat (all scopes in single `standards/` directory with scope prefix in filename)
- **Separation timing**: Deferred to PH002; Model D uses combined files with clear internal heading structure
