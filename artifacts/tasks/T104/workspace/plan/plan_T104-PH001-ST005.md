# PLAN: T104 (CWS) — Phase 1 / Stream `T104-PH001-ST005`: Template & Rules Alignment

## I. EXECUTIVE SUMMARY

**Purpose**: Author the plan-level templates, authoring guideline, and workspace documentation rules updates that codify the planning hierarchy standards (`T104-STD-001`) into reusable governance artifacts. These deliverables serve as: (a) authoring specs for LLM roles producing plan files, and (b) acceptance specs for `T104F (DELIVERABLES)` epic deliverables.

**Non-goal**: This stream does not formalize the activity plan template or LLM_Planner system beyond a high-level skeleton; full formalization is deferred to a future initiative (noted in ST001-SES002).

**Working draft caveat**: All deliverables produced by this stream are initial working drafts. Their details will be confirmed and finalized under epic `T104E (PLAN)` development after `T104-PH001`.

#### Activity AC003: Author `template_workspace_plan_activity.md` (skeleton)

**Activity ID**: `T104-PH001-ST005-AC003`

**Purpose**: Draft a high-level skeleton template for standalone activity plan files. This skeleton defines the structural units (Task as registered unit, Step as informal sub-task execution unit) but does not prescribe Planner-level decomposition rules. Full formalization is deferred to a future initiative.

**Deliverable**:
- `prompt/templates/consultant/workspace/template_workspace_plan_activity.md`

**Scope**:
- In scope:
  - Frontmatter schema (activity-level keys).
  - Required sections: Activity Contract (purpose, deliverable, acceptance criteria, evidence checklist), Task Register (with Task as smallest registered unit), Step guidance (informal, unnumbered sub-bullets within tasks).
  - Trigger rule reference: when this template should be used vs when activity detail stays embedded in stream plan (per `T104-STD-001` CLAUSE-006).
- Out of scope:
  - LLM_Planner decomposition logic.
  - Formal Step ID patterns (deferred; ID slot reserved).
- Working draft: This template is explicitly skeletal. Formalization deferred per ST001-SES002 decision.

**Success Criteria Checklist**:
- [ ] Skeleton defines Task and Step as structural units
- [ ] Trigger rule for standalone activity plan creation is referenced
- [ ] Step is explicitly marked as informal (no ID pattern yet; slot reserved)
- [ ] Template is flagged as "skeleton / working draft"

#### Activity AC004: Author `guideline_workspace_plan.md`

**Activity ID**: `T104-PH001-ST005-AC004`

**Purpose**: Define the authoring rules for plan files across all levels, including depth limits, role boundaries, and decision rules for plan creation.

**Deliverable**:
- `prompt/templates/consultant/workspace/guideline_workspace_plan.md`

**Scope**:
- In scope:
  - Depth limits for task registers per planning level (e.g., phase plan: no task registers; stream plan: thin tasks only; activity plan: full decomposition).
  - Role boundaries per plan level (who may author/modify what — derived from `T104-STD-001` CLAUSE-004).
  - When to link vs when to embed (activity detail in stream plan vs standalone activity plan).
  - When to create a standalone activity plan (trigger rule from CLAUSE-006).
  - When LLM_Planner involvement is required (trigger from CLAUSE-005).
- Out of scope:
  - Template content (that is AC001–AC003).
  - Roadmap authoring rules (those remain in `guideline_workspace_roadmap.md`).

**Success Criteria Checklist**:
- [ ] Depth limits are stated per planning level
- [ ] Role boundaries are defined per plan level with clear ownership
- [ ] Link-vs-embed decision rule is stated
- [ ] Activity plan creation trigger is stated
- [ ] Planner involvement trigger is stated
