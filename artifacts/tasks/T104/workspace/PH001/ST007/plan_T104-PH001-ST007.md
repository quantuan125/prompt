# PLAN: T104 (CWS) — Phase 1 / Stream `T104-PH001-ST007`: Directory Restructuring (Planning + Execution)

## I. EXECUTIVE SUMMARY

**Purpose**: Plan and execute the T104 directory restructuring required to conform to the Client-approved directory and naming conventions (AC000 proposal), including a readiness assessment, migration manifest, script development, and controlled verification.

**Three-phase approach**:
1. **AC000 (Readiness + Manifest)**: Freeze the authority surface, resolve convention gaps identified during readiness analysis, produce a complete current→target migration manifest, and design the scripts approach. Gated on Client review of the manifest before proceeding.
2. **AC001 (Script Development + Execution)**: Develop the migration, validation, and scaffold scripts per the approved manifest and scripts plan. Execute the migration with dry-run evidence before live apply.
3. **AC002 (Verification + Cross-Reference Repair)**: Validate the restructured directory, repair all cross-references, and produce evidence for ST006 Phase 1 exit.
4. **AC003 (Archive Tooling)**: Remediate `archive_manager.py` for Convention 8 conformance (two-tier archive model, dry-run/apply, initiative-scoped archive paths), amend proposal to v3.4.0, and verify scaffold/test coverage. **Completed with GATE-001 PASS (2026-02-23)**.
5. **AC004 (Script Enhancement + P Directory Migration)**: Enhance migration/validation/scaffold scripts for P-initiative compatibility and execute P directory restructuring. Gated on Client approval of dry-run report.
6. **AC005 (T102 Directory Migration)**: Execute T102 directory restructuring including absorption of the legacy `consultant/` directory into T102 root structure. Multi-epic manifests required (T102A, T102B, T102C). Gated on Client approval.

**Scope guardrail (this plan)**: This stream owns the *mechanism* of restructuring (migration manifest, scripts/validators, execution sequencing, and verification evidence). It SHALL NOT reinterpret or renegotiate the approved conventions; it SHALL treat the approved proposal as the normative input until superseded by a program-level `P-STD-004`.

**Authority surface**: `P-STD-004` does not yet exist (`P-PH000-ST001-AC004` status: `planned`). This stream operates in **proposal-seeded mode**: the normative input is `proposal_T104-PH001-ST002-AC000_directory-naming-convention.md` (v3.3.0, 23 DRs Client-approved). If `P-STD-004` is authored before AC001 execution completes, AC001 SHALL reconcile any variance.

**Migration scope**: `prompt/artifacts/tasks/T104/**` (AC001), `prompt/artifacts/tasks/P/**` (AC004), and `prompt/artifacts/tasks/T102/**` (AC005). Scope extended by Client decision on 2026-02-21 to absorb P and T102 migrations into this stream.
