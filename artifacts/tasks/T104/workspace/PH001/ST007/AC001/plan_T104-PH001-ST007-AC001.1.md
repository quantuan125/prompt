### Execution Tracks

The tasks above execute sequentially (single developer). The track groupings below describe the logical purpose of each execution segment and the gate boundaries between them. The `Depends On` column in the Task Register is the enforceable constraint; track groupings are coordination guidance only.

**Track 1 — Manifest Completion (TK001–TK003)**: Produce the complete current→target mapping, define rename rules, and author the machine-readable migration manifest JSON. This is the foundational input for all subsequent tasks.

**Track 2 — Script Hardening + Tests (TK004–TK008)**: Harden both scripts against identified gaps (G1–G11), integrate the reference update pass (DEC003), add retroactive pytest coverage, and ensure required directory creation. Note: TK006 requires both the manifest (TK003) and the hardened migration script (TK004) before it can integrate the reference pass correctly. TK007 requires TK004 and TK005.

**GATE-001**: Client reviews manifest + hardened scripts + passing test suite before dry-run proceeds.

**Track 3 — Dry-Run Execution (TK009–TK011)**: Execute migration in dry-run mode, validate the simulated post-migration state via temp-copy validation (required method), and produce a consolidated evidence report for GATE-002.

**GATE-002**: Client reviews dry-run evidence and approves live execution.

**Track 4 — Live Execution (TK012–TK013)**: Commit git checkpoint, execute live migration, and run post-migration validation including the "no old paths remain" scan (DEC006).
