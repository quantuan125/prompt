# Designing Modular, Dynamic System Prompts for LLM Agents

## Modular Prompt Architecture vs. Monolithic Prompts

Modern LLM-based agents benefit from **protocol-based** system prompts rather than one large monolithic prompt. In a multi-agent setup, each role (Consultant, Planner, Developer, Reviewer, etc.) should have its own system prompt defining role-specific behavior, separate from any global instructions. This *separation of concerns* makes the system more extensible and easier to maintain. Each role’s prompt can follow a standardized template structure (e.g. Role definition, Mandate, Competencies, Execution steps, etc.) to ensure consistency. By contrast, a single giant prompt trying to handle all logic at once is brittle – small changes can ripple unpredictably through the agent’s behavior. Adopting a protocol-like format means the prompt explicitly outlines the “conversation” or sequence of steps the agent should follow (a sort of internal workflow). This approach is akin to treating prompts as **programmatic contracts** or scripts for the agent. For example, multi-agent frameworks like *ChatDev* assign each agent a clear role and interaction protocol, letting them communicate results in a structured way rather than relying on one massive prompt. The takeaway: **modularity** (one prompt per role with well-defined responsibilities) and **protocol-driven design** (defining how an agent should step through its task) lead to more reliable and debuggable agent behavior than a monolithic prompt.

## Layered Context Injection (Static vs. Dynamic Prompts)

A best practice in prompt engineering is to construct the system prompt dynamically by layering in relevant context, instead of using a static, hard-coded prompt. In this project’s design, the final system prompt is assembled at runtime from multiple pieces: the base role prompt plus shared project context and task-specific details. Such **dynamic loading** ensures the agent always has up-to-date information and only the context needed for the current task. For example, common background and definitions can be pulled from a shared JSON (project context) and injected each session, and the current workflow state can inform the prompt of what phase and version it’s working on. This is similar to Retrieval-Augmented Generation (RAG) techniques where external knowledge is fetched and appended to the prompt as needed. By **decoupling knowledge from the prompt**, we avoid hitting context length limits with irrelevant data and reduce hallucinations that arise from stale or overly broad context. In contrast, a static prompt with all possible instructions and info would be lengthy, harder to update, and less focused. Layered injection lets us maintain *session-level context* (project-wide facts or guidelines applicable to all agents), *role-level context* (the base behavior of that agent), and *task-level context* (inputs/outputs specific to the current task) as separate modules that compose into the final system message. This design aligns with the principle that *“prompts are now structured, composable inputs rather than fixed strings”*, enabling more flexible and scalable prompt-driven architectures.

## Manifest-Driven Integrity Verification

To **“trust but verify”** inputs, the system uses manifest files to ensure the agent is working with the correct templates and artifacts. A manifest (like `manifest_standard_request.json`) lists the required components for a given prompt or task template – for example, the structural guideline, procedural guideline, and example sections along with their expected version numbers. The Consultant agent’s system prompt can include an **Integrity Check block** at the start of its execution protocol that directs the agent to verify all required input artifacts are present and match the manifest. This could mean: confirming the latest template version number in use matches the manifest, checking that all sections (structure, process, exemplars) of the `request` template are loaded, and that the `project_context.json` or other config files were applied. If anything is missing or out-of-date, the agent should halt and flag an error instead of proceeding on possibly faulty data (this is a form of *gate halting* to prevent garbage-in). By driving this via a manifest, integrity checks are **externalized** and easy to update – the manifest is the single source of truth for what the prompt needs, so the agent doesn’t rely solely on in-prompt hardcoded instructions. This manifest-driven approach improves reliability by preventing situations where the agent unknowingly uses an old template or omits a section. In essence, the agent is instructed to *“trust”* that it has been given the correct files *but verify* their integrity (presence, version, completeness) before using them. This concept parallels adding verification layers to AI systems as advocated in guardrails frameworks. In practice, the Consultant’s system prompt could contain an initial step like: “**Verify Inputs:** Confirm that the standard request template (v2.2.0) and all its subsections (structure, process guidelines, examples) are loaded per the manifest. If any discrepancy or missing content is found, log an error and do not proceed.” By enforcing such checks, we ensure upstream artifacts are correct, which in a multi-agent chain prevents compounding errors down the line.

## Behavioral Protocols: "Trust but Verify" and Dialogue-First Reasoning

Beyond file integrity, the system prompt should encode behavioral guardrails that implement a *“trust but verify”* mindset in the agent’s reasoning process. This means the agent should not blindly trust every input (including user instructions or outputs from previous agents) without scrutiny. For the Consultant role, this is implemented through a **dialogue-first reasoning** approach: the agent engages the human in an interactive Q\&A to clarify requirements instead of assuming any unspecified details. In the Consultant’s execution steps, notice how it must *“Ask, Don’t Assume”* – formulating clarifying questions for each identified issue. This echoes the ChatDev framework’s strategy of **communicative de-hallucination**, where agents actively request more specifics from each other (or the user) before committing to an answer. By having the prompt explicitly instruct the agent to seek confirmation and details when uncertain, we *verify through dialogue*. This protocol greatly reduces errors: the agent essentially says “I think I understand X, let me double-check via a question” – a direct application of *“trust but verify”* in conversation form.

Another important protocol is **gate halting** for unclear or unsafe situations. The Consultant’s prompt includes an Escalation Protocol that if a requirement remains ambiguous after a couple of attempts, the agent should tag it `[CLARIFICATION NEEDED]` and pause for human intervention. Similarly, if a request conflicts with scope or architecture, the agent must halt (“do not proceed if...”) and escalate. These rules act as *gates* that the agent will not pass until the issue is resolved, thereby ensuring the agent doesn’t charge ahead on faulty premises. All of these behaviors are enforced by clearly writing them into the system prompt as conditional instructions or numbered steps. Essentially, the prompt is programming the agent with a **“Trust but Verify” ethos**: trust the user’s input and prior artifacts as a starting point, but verify every assumption either by asking questions or by checking against documentation. This yields a dialogue-first, verification-driven style of reasoning that is safer and more robust than single-pass answers. Production systems and blogs similarly recommend this approach, noting that adding verification layers and requiring models to validate or justify their outputs leads to more reliable outcomes.

## Embedded Self-Validation and Test Loops

A hallmark of advanced prompt design is giving the agent a chance to **self-evaluate its output** before finalizing. In the system prompt, this can be achieved by embedding a self-validation or test-loop mechanism. Practically, after the agent believes it has completed its task (e.g. drafted the `request` artifact), the prompt can instruct: *“Review the output against all quality criteria and your initial instructions. If any issues are found, fix them before concluding.”* This turns the agent into its own first reviewer. The Consultant’s prompt already lists **Quality Standards** for a good output (e.g. each issue has a Q\&A log, NFRs captured, consistency checks between sections). We can leverage that by having the agent explicitly check each bullet in that list against its draft. This is essentially a **checklist-based self-critique**. For example, the agent could verify: “Did I document NFRs? If not, add an NFR section now.” This embedded loop helps catch omissions or mistakes without needing a human or separate reviewer agent to point them out.

More generally, research on prompting techniques provides many strategies for self-validation:

* *Self-Critique / Reflection:* The agent generates an initial answer, then analyzes it for flaws or missing pieces, and improves it before output. This can be prompted by a simple instruction like “Now critique the above solution and fix any errors,” effectively giving the model a second pass.
* *Self-Refine:* A specific technique where the model iteratively refines its output step by step. In our context, this could be a loop where the agent tackles one section at a time (as the Consultant does per issue) or iteratively improves a draft problem statement until it’s crystal clear.
* *Chain-of-Verification:* The model poses verification questions about its own solution (e.g. *“Is there any part of the plan that might violate the coding standards?”*), answering them to check consistency. An agent could be prompted to simulate a reviewer and ask itself at least one critical question about the output.
* *Multiple Drafts & Comparison:* The system could allow the agent to produce multiple solution variants internally and then compare or vote on the best (some frameworks provide modules like “BestOfN” or ensemble refiners). While not explicitly asked for in this prompt, this is another layer of verification that can be scriptably included (e.g., *“generate two possible approaches and then decide which one meets the criteria better”*).

The system prompt for the Consultant might include a **“Quality Assurance” step** in the Execution Protocol: after assembling the final `Validated Problem Summary`, instruct the agent to perform a final consistency scan (e.g., ensure the summary aligns with the refined issues, ensure no section is empty or unresolved). In more technical agents like the Developer, a self-test loop could involve actually running unit tests via a tool/API call and analyzing the results within the prompt. OpenAI’s function calling can facilitate this by letting the prompt tell the model *when* to invoke a `run_tests` function and how to react to failures. The key is to **embed an expectation of verification**: the agent shouldn’t assume its first solution is always correct. As one Medium article put it, prompts and agent designs are evolving to include guardrails so that every output is checked and possibly corrected before being presented. By having the agent summarize its own performance (e.g., “All checks passed. The artifact meets all criteria.” or listing any remaining concerns), we also gain transparency into the agent’s confidence and reasoning about its solution.

## Proposed System Prompt Structure (Consultant Example)

Taking all these best practices into account, we can outline a concrete design for `consultant_system.md` that serves as a robust, modular system prompt:

* **Role & Mandate:** Begin with a one-line role definition and a short **Core Mandate** section. For example: *“You are a **Senior Technical Consultant** whose mission is to transform a raw client request into a validated technical problem definition.”* This anchors the agent in its identity and primary goal.
* **Key Competencies:** A bullet list of the specific skills/knowledge the role should leverage. The Consultant’s prompt does this (NL deconstruction, inquiry, codebase analysis, etc.). This section conditions the model to utilize those abilities during its reasoning.
* **Execution Protocol:** This is the heart of the system prompt – a numbered sequence of steps the agent must follow for every task. Here we incorporate the **layered context and dynamic behavior**:

  1. *Setup & Ingestion:* e.g. “Acknowledge the task and load all necessary context (project docs, templates)”. This is where an **Integrity Check** sub-step can be inserted: *“Verify that the `request_structural_template.md` template (v2.2.0) and its manifest components are available. If not, report an error.”*.
  2. *Initial Analysis:* e.g. read the raw request and draft a skeleton of issues.
  3. *Refinement Loop:* instruct the agent to iterate through each issue with the client – performing focused research, asking clarifying questions, updating the analysis – until each issue is refined. This implements the **dialogue-first reasoning** and ensures every requirement is verified by client input before finalizing.
  4. *Global Checks:* after individual issues, ask about overall Non-Functional Requirements and scope boundaries (this ensures nothing is missed by focusing only on granular issues).
  5. *Synthesis:* compile the final validated problem summary.
  6. *Self-Validation:* (New) add a step: “Before finalizing, **review the draft** against all quality criteria and the template structure. Ensure each criterion from the Quality Standards section is met; fill in any gaps or correct any inconsistencies.” This is the embedded test-loop where the agent effectively does a self-review.
  7. *Output Registration:* instruct the agent how to save the artifact and mark the task as complete in the workflow (in the current design, a command-line snippet is given to update state).
* **Communication Guidelines:** A section reminding the agent of the tone and style to use when interacting (or producing the artifact content). For the Consultant, this means client-friendly language, asking questions clearly, one issue at a time, etc.. These guidelines act as a soft protocol for how to phrase outputs.
* **Input/Output Specification:** Clearly state what inputs the agent will receive (task\_id, path to raw request) and what output it must produce (the path/pattern of the markdown artifact). This reduces ambiguity – the agent knows exactly where to write its results and in what format.
* **Quality Standards:** Enumerate the success criteria for the output. This list not only communicates to the agent what a good job looks like, but, as discussed, can be used by the agent in a self-check. For example, a criterion “Each issue has a Refinement Summary” means the agent should verify it indeed wrote those summaries.
* **Escalation Protocol:** A final section detailing what to do if things go wrong – when to stop and ask for help. This is vital for safety and aligns with the *gate halting* concept. By explicitly including tags like `[CLARIFICATION NEEDED]` or `[SCOPE CONFLICT]` and instructing their use, the agent is empowered to *fail safe* rather than press on blindly.

All these sections can be modular, possibly assembled from templates themselves. In fact, one can imagine maintaining a **library of prompt modules** (snippets for escalation rules, quality checklists, etc.) that can be included in different role prompts. This is analogous to using a prompt DSL or templating system – rather than writing everything as one-off text, you have reusable blocks. For instance, the *Integrity Check block* and *Self-Validation block* we added could be standardized across agents. A Developer agent’s prompt might reuse those patterns but tuned to its context (e.g., verifying that test data is present via manifest, and a self-test loop where it runs and verifies unit tests). Using a tool like a prompt DSL (domain-specific language) can help here – you could define high-level directives like `{{ load_context(project) }}` or `{{ verify_manifest("standard_request") }}` in a prompt script, which then gets expanded to the proper text or code to execute. This approach has been noted to improve maintainability and reduce errors compared to ad-hoc string concatenation. In fact, frameworks like Microsoft’s *Guidance* or the LMQL language allow developers to programmatically compose prompts with control flow, loops, and conditioned insertions, which could implement things like “for each issue, do Q\&A until resolved” in a more structured way than free-form natural language. However, even without a specialized DSL, our Markdown-based prompt can be organized logically as described, which already imposes a structured, modular form.

## Scaling the Design to Other Agents

The `consultant_system.md` prompt, with the improvements above, can serve as a flagship example for other roles. The same principles apply to designing `planner_system.md`, `developer_system.md`, etc., with adjustments for their unique duties:

* **Shared Global Logic:** All agents load the common project context and definitions (ensuring everyone works from the same global facts and terminologies). This could be handled by a common preamble or simply by each role’s orchestrator code injecting the `shared_definitions.json` content. Either way, consistency is maintained.
* **Role-Specific Behavior:** Each agent’s Core Mandate, Competencies, and Execution Protocol will differ (as defined in documentation) – e.g. the Planner focuses on turning requirements into a step-by-step plan, the Developer writes and documents code, the Reviewer checks quality. But they all follow the *modular prompt template* structure. This means a developer reading any role’s prompt can navigate it easily (sections in a predictable order), and the agents themselves likely benefit from the consistency as well.
* **Integrity Checks via Manifest:** Similar to the request template manifest, we can have manifests for plan templates, development note templates, etc. The Planner agent’s prompt can verify it has the latest planning guidelines and template before generating a plan. The Developer agent can verify that it has the correct version of coding standards or test cases available. Embedding a brief check at the start of each role’s execution ensures each phase uses the correct inputs. For example, the Developer might check that the plan it received is marked “approved” and matches the version in the state file – if not, it should halt and perhaps ask the Planner or human to resolve the discrepancy. This prevents an agent from building on possibly invalid output from the previous phase, embodying a *“trust but verify” inter-agent protocol*.
* **Behavioral Protocols:** Each role’s prompt can enforce *“trust but verify”* in role-appropriate ways. The Planner, for instance, should trust the Consultation output but verify it by cross-checking requirements – if something in the consultation seems unclear, the Planner could be prompted to add a note or ask a clarification (maybe via a feedback loop to the Consultant or directly to the human). The Developer agent should similarly double-check if the plan’s instructions are feasible; if a step in the plan seems likely to produce an error, the Developer might loop back to propose an adjustment (this could be an advanced feature where agents can feed issues back upstream – effectively implementing a feedback mechanism in the multi-agent workflow). While such cross-agent dialogue is complex, the system prompt can at least make the Developer aware of its scope: e.g. “If you encounter an obvious error in the plan instructions, flag it `[PLAN CONFLICT]` and stop rather than writing incorrect code.”
* **Execution Loops and Self-Testing:** The Developer role naturally lends itself to a test loop – after writing code, it can be instructed to simulate running it or to check it for errors. If we integrate an automated test harness (via tools), the Developer’s prompt can include steps like “Run the test suite. If tests fail, go back and fix the code (iterate) up to 2 times.” This would be facilitated by function calling where the model chooses to invoke a `run_tests` function and gets results. The *AutoGPT*-style agents use this kind of loop where the agent can critique its attempt and try again, which we can mirror in the prompt as a set of iterative steps. The Reviewer agent, on the other hand, might not have a loop in the same way, but its prompt can contain a checklist (much like Quality Standards) to systematically verify each aspect (requirements coverage, code quality, documentation, etc.) and then produce a verdict. Essentially, we inject a *mini quality assurance process* into each agent’s prompt.
* **Consistent Protocol and Language:** Because all agents are part of one workflow, their prompts should reference the overall process to stay aligned. Indeed, the role prompts are expected to reference the master workflow documentation (the `prompt_main.md`) to ground themselves. This is a reminder in the Validator script that each role prompt must mention the main workflow for context. In practice, including a line like “*(Refer to the overall workflow in prompt\_main.md for where you hand off next)*” in the Execution Protocol keeps the agent aware of the bigger picture. It also helps ensure that if the process changes, the reference alerts the prompt designer to update each role accordingly.

By designing each agent’s system prompt with **modular sections and inserted dynamic context**, and by reusing **verification and self-check modules**, we create a cohesive multi-agent system. The prompts act almost like a relay baton passed between agents: each agent trusts that the previous did its job, but verifies crucial details before carrying the task forward, and each adds its specialized contribution in a controlled, step-by-step manner. This protocol-based modular prompt strategy makes the overall system more **scalable** (we can introduce new roles or swap out templates with minimal changes) and **robust** (errors are caught early by agents themselves). As one expert noted, treating prompts as an **architectural component** – with modularity, testability, and standardization – is key to building reliable AI agents at scale. The `consultant_system.md` prompt, once outfitted with these advanced techniques, will serve as a template for other agents’ prompts. Future agents can follow the same pattern in `prompt_main.md`’s framework, plugging in their specific logic but adhering to the layered context injection, verification steps, and self-evaluation loops that define this *next-generation* prompt engineering approach.
